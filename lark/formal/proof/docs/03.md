
## Terms and Values - The Two Worlds

After the first-read tour you know that `Term` and `Val` both exist and that
`nbe_eval` crosses from one to the other. This chapter goes deeper into why
the two representations exist and what each one is optimised for. Understanding
this distinction is the key to reading the entire codebase.



### 1. Why two representations?

Every expression in this system lives in two worlds simultaneously.

*`Term`* is the syntactic world. It is what the parser produces and what the
printer turns back into text. A `Term` is a tree of `TM_*` nodes built by the
`tm_*` constructor functions in `term.c`. It represents *source code*: the
structure of what was written, not the result of running it.

*`Val`* is the semantic world. It is the *meaning* of a term - the result of
running it as far as possible. A `Val` is a `VL_*` node built by the `vl_*`
constructors. The type checker works entirely in this world: it computes types
as values and compares them as values.

Each world is optimised for its purpose:

- `Term` is built for *writing* and *parsing*: de Bruijn indices make
  alpha-equivalence trivial (no renaming ever needed). The tree shape mirrors
  the surface grammar, so the parser can construct `Term`s directly and the
  printer can recover source from them.

- `Val` is built for *computing*: closures make substitution free. Lambda
  bodies are stored unevaluated alongside the environment that was current
  when the lambda was created. De Bruijn *levels* (rather than indices) make
  comparison free - two values share the same variable if and only if their
  levels match, with no index adjustment needed when going under binders.

The boundary between them is always explicit:

- `nbe_eval(env, Term) â†’ Val` crosses from syntax to semantics
- `nbe_quote(depth, Val) â†’ Term` crosses back



### 2. De Bruijn indices vs levels

Both `Term` and `Val` use nameless variable representations, but they use
*different* nameless representations. This is the single most confusing aspect
of the codebase until you see the reason.

*In `Term`s:* variables are *de Bruijn indices*. An index is a count of how
many enclosing binders to skip. Index 0 refers to the nearest enclosing
binder; index 1 skips one binder; and so on.

*In `Val`s:* variables are *de Bruijn levels*. A level is an absolute count
from the *outside in*. Level 0 is the outermost bound variable encountered
so far; level 1 is the next one in; and so on.

Work through a concrete example: `\A. \x. A`.

As a `Term`:
```
TM_LAM("A", TM_LAM("x", TM_VAR(1)))
```
Inside the inner lambda, `A` is index 1 (skip past `x` to reach `A`), and
`x` would be index 0.

Now evaluate this term starting at depth 0. The outer lambda binds `A` at
level 0; the inner lambda binds `x` at level 1. When we open the inner body
with a fresh neutral variable, we are at depth 2. The formula to convert a
level back to an index is:

```
index = depth - level - 1
```

- `A` has level 0; at depth 2: index = 2 âˆ’ 0 âˆ’ 1 = 1. âœ“
- `x` has level 1; at depth 2: index = 2 âˆ’ 1 âˆ’ 1 = 0. âœ“

In `nbe_quote`, this conversion is:

```c
case VL_NEUTRAL:
    return tm_var(a, depth - v->neutral.lvl - 1);
```

Why maintain two different representations? Indices are convenient for
substitution-free evaluation: when you push an argument onto the front of
`env`, index 0 automatically refers to it without updating anything else.
Levels are convenient for comparison: to check whether two neutral values
are the same variable, compare their levels directly. No context, no
adjustment needed.



### 3. Closures - why `pi.cod` is a `Term`, not a `Val`

Open `term.h` and find the fields of `VL_PI` in the `Val` union:

```c
struct { char *name; Val *dom; Env *env; Term *cod; } pi;
```

The domain `dom` is a `Val`, but the codomain `cod` is stored as a `Term`
together with a captured `env`. Why not evaluate `cod` immediately?

A dependent Pi type `Î (x:A). B(x)` has a codomain `B` that can *mention*
`x`. Until the actual argument arrives, we cannot evaluate `B(x)` - `x`
is unknown. So we store `cod` as a closure: a `(env, Term)` pair. When an
argument value `v` arrives, we open the codomain:

```c
nbe_eval(a, env_cons(a, v, pi->env), pi->cod)
```

This is closure application. No substitution, no tree traversal - just one
call to `nbe_eval` with an extended environment.

If we had pre-evaluated `cod` to a `Val`, we would be stuck: `Val` can
contain free variables as `VL_NEUTRAL` nodes, but there is no general
substitution operation on `Val`. The closure representation sidesteps the
problem entirely by deferring evaluation until the argument is known.

The same pattern appears in `VL_LAM`, `VL_SIG`, and `VL_W`: anywhere a
binder's body can depend on the bound variable, the body is stored as
`(Env, Term)` rather than as a `Val`.



### 4. Neutral values and spines

Not every term reduces to a constructor. If the scrutinee of an eliminator
is a variable - something whose value is unknown - no reduction rule can
fire. The computation is *stuck*.

The system represents stuck computations as *neutral values*:

```c
VL_NEUTRAL(lvl, spine)
```

`lvl` is the de Bruijn level of the variable at the head of the stuck term.
`spine` is a linked list of eliminators accumulated on that head, stored as
`SpineNode` entries.

Examples of neutral values (assuming `f` and `x` are free variables at
levels `lf` and `lx` respectively):

- `f zero` with `f : Nat â†’ Nat` free â†’
  `VL_NEUTRAL(lf, [SP_APP{VL_ZERO}])`

- `natrec P z s x` with `x : Nat` free â†’
  `VL_NEUTRAL(lx, [SP_NATREC{P,z,s}])`

- `fst p` with `p` free â†’
  `VL_NEUTRAL(lp, [SP_FST])`

Reading a spine: the head is at level `lvl`; the spine records what has
been applied to it, from innermost to outermost. To reconstruct the original
term, `nbe_quote` traverses the spine and builds `TM_*` nodes:

```c
/* sketch of spine replay in nbe_quote */
Term *t = tm_var(a, depth - neutral->lvl - 1);
for (SpineNode *s = neutral->spine; s; s = s->next) {
    switch (s->tag) {
    case SP_APP:    t = tm_app(a, t, nbe_quote(a, depth, s->arg)); break;
    case SP_NATREC: t = tm_natrec(a, ..., t); break;
    /* ... */
    }
}
return t;
```

Neutral values are what makes open-term evaluation work. During type
checking, variables whose values are unknown appear as fresh neutrals pushed
into the environment. Eliminators applied to them accumulate spines, and
`nbe_quote` replays those spines back to readable syntax.



### 5. The conversion invariant

`conv(a, depth, u, v)` in `check.c` tests whether two values are
definitionally equal. Its key invariant: *it is always called with
already-evaluated `Val`s*. It never calls `nbe_eval` itself.

This is possible because the type checker evaluates types as it builds
them. By the time `conv` is called, both sides have been fully evaluated
by the surrounding `infer`/`check` calls. `conv` only needs to compare
them structurally.

The cases:

- *Matching constructor tags:* recurse on subterms.
  `VL_SUCC u` = `VL_SUCC v` iff `conv(u, v)`.

- *`VL_LAM` on either side (eta):* apply both sides to a fresh variable at
  `depth`, then compare the results at `depth + 1`. This ensures `\x. f x`
  and `f` are treated as equal (eta-equivalence).

- *`VL_NEUTRAL`:* compare the head levels first, then compare each spine
  element pairwise. Two neutral values are equal iff they are the same
  variable with the same eliminators applied.

- *Mismatching tags:* not convertible; return 0.

The eta case is particularly important. The type checker uses `conv` to
verify that a term has the expected type, and types can be eta-equivalent
without being identical. Without the eta case, functions would need to be
written in fully eta-expanded form to typecheck.

Because `conv` never evaluates, *definitional equality reduces to structural
equality after evaluation*. This is the payoff of NbE: one `nbe_eval` pass
puts both sides into normal form, and then `conv` is a fast structural walk.
