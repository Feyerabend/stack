
## First Read - Tracing One Expression End to End

We will follow a single expression through every layer of the system:

```
((\x. x) : Nat → Nat) zero
```

This is the identity function, annotated with type `Nat → Nat`, applied to
`zero`. The result should be `zero`. The type should be `Nat`.

There are no tricks here. This expression exercises parsing, type checking via
bidirectional typing, NbE evaluation, and printing - the complete pipeline.
Every function we touch is something you will use constantly when extending the
system.

Start with `code/core/` open in an editor.



### Stage 1 - The REPL loop (`main.c`)

The user types the expression. The REPL function `run_infer` takes over:

```c
/* main.c - stripped to essentials */
static void run_infer(Arena *a, const char *src) {
    Term *t = parse(a, src);
    if (!t) return;
    /* No holes, so skip elaboration */
    Term *nf = nbe_nf(a, t);
    Val  *ty = infer(a, 0, NULL, NULL, t);
    /* print ty and nf */
}
```

Three things happen: parse, type-check, normalise. They are independent - the
type checker and normaliser do not call each other at the top level.

`Arena *a` is the memory arena. All allocation goes through it. There is no
`free` in this codebase. When you are done with a computation you discard the
entire arena.



### Stage 2 - Parsing (`parse.c`)

`parse(a, src)` calls the recursive-descent parser. The expression
`((\x. x) : Nat → Nat) zero` is parsed as follows.

The outer structure is a function application. The parser sees a parenthesised
group--an annotation--applied to an argument.

The *annotation* `(\x. x) : Nat → Nat` has two parts:
- the term: `\x. x`
- the type: `Nat → Nat`

The *application* applies that annotation to `zero`.

The resulting `Term` tree:

```
TM_APP
├── fun: TM_ANN
│     ├── term: TM_LAM("x")
│     │         └── body: TM_VAR(0)
│     └── type: TM_PI("_")
│               ├── dom: TM_NAT
│               └── cod: TM_NAT          ← index 0 would mean "_", unused
└── arg: TM_ZERO
```

*De Bruijn indices.* Variables are numbers, not names. `TM_VAR(0)` inside
`TM_LAM("x", ...)` refers to the closest enclosing binder - which is `x`.
The name `"x"` is stored only for printing; it plays no role in evaluation
or checking.

The type `Nat → Nat` is `Π(_:Nat). Nat`. The codomain `TM_NAT` does not use
the bound variable (it is a constant type), but it is still written under a
binder in the `Term` representation because all Pi codomains are open terms.

*The `term_has_holes` check.* Before going further, `main.c` calls
`term_has_holes(t)`. This walks the term looking for `TM_HOLE`. There are none,
so elaboration is skipped entirely. Elaboration only runs when you write `_`.



### Stage 3 - Type inference (`check.c: infer`)

The goal is to answer: *what type does this term have?*

```c
Val *ty = infer(a, 0, NULL, NULL, t);
```

Arguments:
- `a` - arena
- `0` - current depth (de Bruijn level counter for fresh variables)
- `NULL` - type context (TCtx): list of variable types, innermost first
- `NULL` - environment (Env): list of variable values, innermost first
- `t` - the term

`infer` is a large switch on `t->tag`. The outer term is `TM_APP`, so we enter:

```c
case TM_APP: {
    Val *fty = infer(a, depth, tctx, env, t->app.fun);   /* [A] */
    if (fty->tag != VL_PI) { error; return NULL; }
    check(a, depth, tctx, env, t->app.arg, fty->pi.dom); /* [B] */
    Val *argv = nbe_eval(a, env, t->app.arg);            /* [C] */
    return nbe_eval(a,                                   /* [D] */
        env_cons(a, argv, fty->pi.env), fty->pi.cod);
}
```

#### [A] Infer the type of the function

The function is `TM_ANN(TM_LAM("x", TM_VAR(0)), TM_PI("_", TM_NAT, TM_NAT))`.

```c
case TM_ANN: {
    Val *tty = infer(a, depth, tctx, env, t->ann.type);   /* infer the annotation */
    /* check it is a universe */
    Val *tyv = nbe_eval(a, env, t->ann.type);             /* evaluate it */
    check(a, depth, tctx, env, t->ann.term, tyv);         /* check term against it */
    return tyv;
}
```

Step by step:

1. *Infer the type of the annotation* `Nat → Nat`.  
   This is `TM_PI("_", TM_NAT, TM_NAT)`. Entering the `TM_PI` case:
   ```c
   Val *dty = infer(a, depth, tctx, env, t->pi.dom);  /* infer Nat → VL_UNI(0) */
   Val *domv = nbe_eval(a, env, t->pi.dom);            /* eval Nat → VL_NAT    */
   /* open the codomain with a fresh variable at level 0 */
   Val *fresh = vl_neutral(a, 0, NULL);
   TCtx ext = { "_", domv, tctx };
   Val *cty = infer(a, 1, &ext, env_cons(a,fresh,env), t->pi.cod);
   /* infer Nat (in extended context) → VL_UNI(0) */
   return uni_at(a, imax(0, 0));  /* → VL_UNI(0) */
   ```
   `Nat → Nat` has type `Type` (universe level 0). ✓

2. *Evaluate the annotation* `Nat → Nat` into a Val:
   ```
   nbe_eval(env=[], TM_PI("_", TM_NAT, TM_NAT))
   → vl_pi("_", VL_NAT, env=[], TM_NAT)
   ```
   `tyv` = `VL_PI("_", dom=VL_NAT, env=[], cod=TM_NAT)`

   Notice: the codomain is kept as a closed `Term` plus a captured `Env`. This
   is how Pi types represent dependent codomains without substitution - the cod
   is only evaluated when you apply the Pi to an argument.

3. *Check the term* `\x. x` against `tyv`.

   `check` is called. The TM_LAM case:
   ```c
   if (t->tag == TM_LAM) {
       if (ty->tag != VL_PI) { error; return 0; }
       Val *fresh = vl_neutral(a, depth, NULL);          /* fresh var at level 0 */
       Val *codv  = nbe_eval(a,                          /* open the codomain */
           env_cons(a, fresh, ty->pi.env), ty->pi.cod);
       TCtx ext = { t->lam.name, ty->pi.dom, tctx };     /* extend type context */
       return check(a, depth+1, &ext,                    /* check body */
           env_cons(a, fresh, env), t->lam.body, codv);
   }
   ```

   - `ty->pi.dom` = `VL_NAT` - the type of `x`
   - `fresh` = `VL_NEUTRAL(0, NULL)` - a stand-in for `x` during checking
   - `codv` = `nbe_eval([fresh], TM_NAT)` = `VL_NAT` - the return type
   - Extend context: `ext = {name="x", type=VL_NAT, next=NULL}`
   - Check body `TM_VAR(0)` against `VL_NAT` at depth 1:

   ```c
   /* TM_VAR(0) is not TM_LAM, not TM_FIX, not TM_PAIR, etc.
      Fall through to infer + conv. */
   Val *ity = infer(a, 1, ext, env=[fresh], TM_VAR(0));
   /* TM_VAR case: tctx_lookup(ext, 0) = VL_NAT */
   ity = VL_NAT;
   conv(a, 1, VL_NAT, VL_NAT)  → 1  ✓
   ```

   `check` returns 1 (success). The lambda is well-typed. ✓

4. *Return* `tyv` = `VL_PI("_", VL_NAT, [], TM_NAT)` - the type of the
   annotated expression.

So `fty` = `VL_PI("_", VL_NAT, [], TM_NAT)`.

#### [B] Check the argument

```c
check(a, 0, NULL, NULL, TM_ZERO, fty->pi.dom)
```

`fty->pi.dom` = `VL_NAT`. `TM_ZERO` is not a LAM/FIX/PAIR - falls through to
infer+conv:

```c
infer(a, 0, NULL, NULL, TM_ZERO)
→  case TM_ZERO: return vl_nat(a);   /* VL_NAT */

conv(a, 0, VL_NAT, VL_NAT)  → 1  ✓
```

#### [C] Evaluate the argument

```c
Val *argv = nbe_eval(a, NULL, TM_ZERO)
→  case TM_ZERO: return vl_zero(a);   /* VL_ZERO */
```

#### [D] Compute the return type

The Pi codomain is `TM_NAT` under environment `[]`. We instantiate it with the
argument value:

```c
nbe_eval(a, env_cons(VL_ZERO, []), TM_NAT)
→  case TM_NAT: return vl_nat(a);   /* VL_NAT */
```

`TM_NAT` ignores its environment (it has no free variables), so the result is
just `VL_NAT`. In a dependent type like `Vec A n`, the codomain *would* look up
the argument in the environment.

*`infer` returns `VL_NAT`.* The whole expression has type `Nat`. ✓



### Stage 4 - Normalisation (`eval.c: nbe_eval, nbe_vapp`)

```c
Term *nf = nbe_nf(a, t);
```

`nbe_nf` = `nbe_quote(a, 0, nbe_eval(a, NULL, t))`. Two steps: evaluate, then
quote back.

#### Evaluation

```c
nbe_eval(a, env=NULL, TM_APP(TM_ANN(TM_LAM("x",TM_VAR(0)), ...), TM_ZERO))
```

```c
case TM_APP:
    fun_val = nbe_eval(a, NULL, TM_ANN(TM_LAM("x",TM_VAR(0)), ...))
    arg_val = nbe_eval(a, NULL, TM_ZERO)
    return nbe_vapp(a, fun_val, arg_val)
```

*`fun_val`:*
```c
case TM_ANN:
    return nbe_eval(a, NULL, TM_LAM("x", TM_VAR(0)))
→  case TM_LAM:
    return vl_lam("x", env=NULL, body=TM_VAR(0))
```

The annotation is erased by evaluation - `TM_ANN` simply evaluates its term
part. The type part is only used by the type checker, never by the evaluator.

`fun_val` = `VL_LAM("x", env=NULL, body=TM_VAR(0))`

*`arg_val`:*
```c
case TM_ZERO: return vl_zero(a)
```
`arg_val` = `VL_ZERO`

*`nbe_vapp`:*
```c
Val *nbe_vapp(Arena *a, Val *fun, Val *arg) {
    switch (fun->tag) {
    case VL_LAM:
        return nbe_eval(a, env_cons(a, arg, fun->lam.env), fun->lam.body);
    ...
    }
}
```

```c
nbe_vapp(VL_LAM("x", NULL, TM_VAR(0)), VL_ZERO)
→  nbe_eval(a, env=[VL_ZERO], TM_VAR(0))
→  case TM_VAR(0): env_lookup([VL_ZERO], 0, 0)
→  VL_ZERO
```

*Result of evaluation:* `VL_ZERO`

This is beta reduction. No rewrite rules, no explicit substitution - just
evaluating `TM_VAR(0)` in an environment where position 0 holds `VL_ZERO`.

#### Quoting

```c
nbe_quote(a, 0, VL_ZERO)
→  case VL_ZERO: return tm_zero(a)
```

*`nf`* = `TM_ZERO`. Printed as `"zero"`. ✓



### Stage 5 - The two worlds

Every expression lives in two worlds simultaneously. Understanding the
distinction is the key to reading this codebase.

| World     | Type   | Created by                      | Represents                |
|-----------|--------|---------------------------------|---------------------------|
| Syntax    | `Term` | parser, `tm_*` constructors     | Source code as a tree     |
| Semantics | `Val`  | `nbe_eval`, `vl_*` constructors | Meaning - computed values |

The boundary between them:
- *`nbe_eval(env, Term) → Val`* crosses from syntax to semantics
- *`nbe_quote(depth, Val) → Term`* crosses back

The type checker lives in semantics (`Val`): it computes and compares types as
values, not as syntax trees. Definitional equality (`conv`) on `Val` is just
structural comparison - it works because evaluation already did the computation.

The *`Env`* is a linked list mapping de Bruijn indices to `Val`s. When you
evaluate `TM_VAR(0)`, you look up index 0 in `env`. When you enter a lambda
body, you push the argument value onto the front of `env`.

The *`TCtx`* (type context) is a parallel list mapping de Bruijn indices to
`Val` *types*. The type checker uses `TCtx`; the evaluator uses `Env`. They
grow in lockstep: whenever `check` extends `TCtx` with a new variable's type,
it extends `Env` with a fresh neutral `VL_NEUTRAL(depth, NULL)` to stand in for
that variable's unknown value.



### Stage 6 - What `depth` counts

`depth` starts at 0 at the top level. Every time we go under a binder (enter a
lambda body for checking, or open a Pi codomain), we increment it.

`vl_neutral(a, depth, NULL)` creates a fresh variable at the current level.
Because `depth` strictly increases, each fresh variable gets a unique level and
can be told apart from all others.

When `quote` sees `VL_NEUTRAL(lvl, spine)` and needs to turn it back into a de
Bruijn *index*, it computes `depth - lvl - 1`. At depth 3, level 0 → index 2
(the outermost variable), level 2 → index 0 (the innermost).

This is why the two representations differ: `Term` uses *indices* (relative,
innermost = 0), `Val` uses *levels* (absolute, outermost = 0). Indices are
convenient for substitution-free evaluation; levels are convenient for
comparison (two values share the same fresh variable iff their levels match).



### What to look at next

After reading this document, open these files in order:

1. `core/term.h` - All `TM_` and `VL_` tags, plus all constructor functions.
   Skim it once to know what exists; return to it as a reference.

2. `core/eval.c` - `nbe_eval` (big switch), `nbe_vapp` (beta), the eliminators
   (`nbe_vnatrec`, `nbe_vboolrec`, etc.). Each eliminator has the same pattern:
   fire on a constructor, accumulate on a neutral.

3. `core/check.c` - `infer` (big switch), `check` (small, delegates to
   `infer`+`conv` for most cases), `conv` (structural equality on `Val`).

4. `core/parse.c` - `parse_term`, `parse_atom`. The de Bruijn conversion is in
   `name_lookup` + the `NameCtx` threading through recursive calls.

5. `core/elab.c` - `elab_infer`, `elab_check`, `elab_unify`. Only relevant
   when you write `_`. Skip this on a first reading.



### A map of the calls in this example

```
run_infer("((\x. x) : Nat → Nat) zero")
│
├── parse(a, src)
│   └── recursive descent → TM_APP(TM_ANN(TM_LAM, TM_PI), TM_ZERO)
│
├── infer(a, 0, NULL, NULL, t)
│   └── TM_APP case
│       ├── infer fun (TM_ANN)
│       │   ├── infer annotation type (TM_PI)    → VL_UNI(0)
│       │   ├── nbe_eval annotation type         → VL_PI("_", VL_NAT, [], TM_NAT)
│       │   └── check term (TM_LAM) against VL_PI
│       │       └── check body (TM_VAR 0) against VL_NAT
│       │           └── infer TM_VAR(0)           → VL_NAT  ✓
│       ├── check arg (TM_ZERO) against VL_NAT
│       │   └── infer TM_ZERO                    → VL_NAT  ✓
│       ├── nbe_eval arg (TM_ZERO)               → VL_ZERO
│       └── nbe_eval codomain with [VL_ZERO]     → VL_NAT
│
└── nbe_nf(a, t)
    ├── nbe_eval(NULL, TM_APP(...))
    │   ├── nbe_eval fun → VL_LAM("x", NULL, TM_VAR(0))
    │   ├── nbe_eval arg → VL_ZERO
    │   └── nbe_vapp(VL_LAM, VL_ZERO)
    │       └── nbe_eval([VL_ZERO], TM_VAR(0))   → VL_ZERO
    └── nbe_quote(0, VL_ZERO)                    → TM_ZERO
```
