
## Normalisation by Evaluation

This chapter explains what NbE is, why this system uses it instead of term
rewriting, and how evaluation, quoting, and eliminators interact in detail.
Read chapter 03 first - the `Term`/`Val` distinction is assumed throughout.



### 1. The alternative: substitution-based reduction

The naive approach to normalisation is *substitution-based reduction*. When
you see a beta redex `(\x. t) s`, walk the tree `t` and replace every
occurrence of `x` with `s`, adjusting de Bruijn indices of any variables
inside `s` each time you go under a binder.

This works, but has three compounding problems:

*Every beta step traverses the whole body.* A term like `(\x. long_body) s`
requires a complete walk of `long_body` even if `x` does not appear in it.

*Nested lambdas require index shifting.* Substituting `s` into a term that
contains further binders requires incrementing all free-variable indices in
`s` each time you descend under a new binder. This is `lift` or `shift` in
textbook presentations.

*No sharing.* If `x` appears three times in `t`, then `s` is copied three
times into the result. A large `s` can make the output exponentially larger
than the input.

For a type checker that calls normalisation inside every `infer` call for
dependent Pi types, this cost accumulates quickly.



### 2. The NbE approach

NbE avoids substitution by splitting normalisation into two passes:

```
nbe_eval  : Term × Env → Val    (syntax → semantics, one traversal)
nbe_quote : depth × Val → Term  (semantics → syntax, one traversal)
```

*Evaluation* (`nbe_eval`) traverses the term once, building `Val` nodes.
It never copies subterms. When it encounters a lambda, it wraps the body
in a closure - a `(env, body)` pair - without touching the body:

```c
case TM_LAM:
    return vl_lam(a, t->lam.name, env, t->lam.body);
```

Beta reduction happens in `nbe_vapp`:

```c
case VL_LAM:
    return nbe_eval(a, env_cons(a, arg, fun->lam.env), fun->lam.body);
```

The argument `arg` is pushed onto the front of the captured environment.
The body is then evaluated with this extended environment. When evaluation
reaches a `TM_VAR(i)` inside the body, it looks up index `i`:

```c
case TM_VAR:
    return env_lookup(env, t->var.idx, depth);
```

This is substitution - but it happens at look-up time, one variable at a
time, only for variables that are actually used. No upfront tree traversal.
No index shifting.

*Quoting* (`nbe_quote`) traverses the `Val` once, rebuilding a `Term`. The
type checker does not quote; it works with `Val`s directly. Quoting is only
called by the REPL (to print a result) or by `nbe_nf` (for full
normalisation).

The two-pass structure means beta reduction is free during evaluation (it
is just environment look-up), and quoting is only paid for when you
actually need a printable result.



### 3. Why quote?

If evaluation produces a `Val`, why reconstruct a `Term` at all?

*Printing.* The REPL prints `Term`s. To print the result of evaluating an
expression, it must be quoted back to syntax.

*Eta-long normal forms.* A `VL_LAM` is a function waiting to be applied.
To quote it, you cannot simply emit a `TM_LAM` node without knowing the
body. `nbe_quote` handles `VL_LAM` by:

1. Creating a fresh variable at `depth`: `VL_NEUTRAL(depth, NULL)`
2. Applying the lambda closure to this fresh variable via `nbe_vapp`
3. Quoting the result at `depth + 1`

```c
case VL_LAM: {
    Val  *fresh = vl_neutral(a, depth, NULL);
    Val  *body  = nbe_vapp(a, v, fresh);
    Term *qt    = nbe_quote(a, depth+1, body);
    return tm_lam(a, v->lam.name, qt);
}
```

This produces the *eta-long* form: if `v` is a value representing the
function `f`, this emits `\x. f x` rather than just `f`. Eta-expansion
matters for `conv`: when comparing `VL_LAM` against another value,
`conv` applies both to a fresh variable and compares the bodies, which
requires eta-long results to be well-defined.

*Neutral spine replay.* Quoting a `VL_NEUTRAL` rebuilds the stuck term by
replaying the spine: first emit the head variable as `TM_VAR`, then wrap
it in `TM_*` applications for each spine element (each sub-`Val` in the
spine is recursively quoted):

```c
case VL_NEUTRAL: {
    Term *t = tm_var(a, depth - v->neutral.lvl - 1);
    for (SpineNode *s = v->neutral.spine; s; s = s->next) {
        switch (s->tag) {
        case SP_APP:    t = tm_app(a, t, nbe_quote(a, depth, s->arg)); break;
        case SP_NATREC: t = tm_natrec(a, ..., t); break;
        /* ... */
        }
    }
    return t;
}
```

This reconstructs `natrec P z s x` or `f zero` from the accumulated spine,
yielding a `Term` that can be printed or compared structurally.



### 4. Eliminators on neutrals - the firing rule

Every eliminator in `eval.c` follows the same three-case pattern. Here it
is for `nbe_vnatrec`:

```c
Val *nbe_vnatrec(Arena *a, Val *motive, Val *base, Val *step, Val *n) {
    if (n->tag == VL_ZERO)
        return base;
    if (n->tag == VL_SUCC)
        return nbe_vapp(a,
            nbe_vapp(a, step, n->succ),
            nbe_vnatrec(a, motive, base, step, n->succ));
    if (n->tag == VL_NEUTRAL)
        return vl_neutral(a, n->neutral.lvl,
            spine_cons(a, SP_NATREC, motive, base, step, n->neutral.spine));
    /* unreachable for well-typed terms */
    abort();
}
```

*Case 1: Constructor.* The scrutinee is `VL_ZERO` or `VL_SUCC`: the
reduction rule fires and the result is computed directly. For `VL_SUCC`,
the step function is applied to the predecessor and to the recursive result
on the predecessor.

*Case 2: Neutral.* The scrutinee is headed by a free variable. No rule
can fire. The eliminator is pushed onto the spine and a new neutral is
returned, carrying the original spine extended with this `natrec`.

*Case 3: Anything else.* Unreachable for a well-typed term. Only `VL_ZERO`,
`VL_SUCC`, or a neutral can appear as the scrutinee of `natrec` - if
something else appears, the term was ill-typed to begin with.

Every eliminator in the codebase follows this pattern: `nbe_vboolrec`,
`nbe_vcase` (for sums), `nbe_vwrec` (for W-types), `nbe_vindrec` (for
inductive families). Once you understand `nbe_vnatrec`, you have understood
all of them. The three cases - fire on a constructor, accumulate on a
neutral, abort otherwise - are the complete story.

Spine accumulation is what makes open-term type checking possible. The type
of `natrec P z s x` (with `x` a free variable of type `Nat`) is still
computable: it is `P x`, represented as `nbe_vapp(motive, VL_NEUTRAL(lx, ...))`.
The motive is applied to the neutral scrutinee, producing a neutral result.
The type checker can work with this and continue.



### 5. Checking normalises on demand

The type checker calls `nbe_eval` exactly when it needs a value - not
eagerly, and not speculatively.

The key call sites in `check.c`:

- `TM_ANN`: the annotation type is evaluated so it can be passed to `check`
  as a `Val` target.
- `TM_APP`: the argument is evaluated so it can be substituted into the Pi
  codomain via `env_cons`.
- Opening a Pi codomain: `nbe_eval(a, env_cons(a, fresh, pi->env), pi->cod)`
  opens the codomain for a fresh neutral variable.

`nbe_nf` (full normalisation to a `Term`) is called once per expression
by the REPL, only to print the result. It is not called by the type checker.

This matters in practice: checking that `plus two three : Nat` never
requires computing the value `five`. The type checker confirms `Nat` is
the return type of `plus` applied to two arguments, and stops there.
The REPL then separately calls `nbe_nf` to print `succ (succ (succ (succ
(succ zero))))`.

Separating checking from normalisation also means type checking is safe
for terms that diverge under evaluation: a fixpoint might loop when run,
but its type can still be checked if the type itself is well-formed and
does not require evaluating the fixpoint.

`Nat → Nat`. Test with `double (succ (succ zero))`. How many lines does
