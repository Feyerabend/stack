
## Orientation - What This Is and How to Read It

A working proof checker for a dependent type theory, written in C. It
implements Martin-Löf Type Theory (MLTT) extended with features from Homotopy
Type Theory (HoTT). The codebase is around 4 000 lines across a dozen files,
and every line is directly connected to the mathematical concepts it encodes.

The system has two layers:

*Core* (`code/core/`) - The mathematical kernel. A parser, a bidirectional
type checker, a normaliser, and an elaborator that fills in arguments you left
as `_`. The REPL is `lcore`.

*Lang* (`code/lang/`) - A call-by-need graph reducer using the same surface
syntax and type checker. The REPL is `llang`. You can ignore this layer
entirely until you are interested in evaluation strategy.



### Building and running

```
cd code/core && make
./lcore
```

The REPL accepts one expression per line:

```
> succ (succ zero)
  parsed : succ (succ zero)
  normal : succ (succ zero)
> :type Π(A : Type). A → A
  type : Type_1
> :let id : Π(A : Type). A → A = \A x. x
> :type id _ zero
  type : Nat
```

Commands: `:type e` - infer and print the type of `e`. `:conv a b` - test
definitional equality. `:let name : type = term` - add a global definition.

For the lang layer: `cd code/lang && make && ./llang`, then
`:load "lib/prelude.lam"` to load the standard library.



### What "types as propositions" means here

Every type in this system is simultaneously a proposition. A term `t : A` is
both a program of type `A` and a proof that `A` is inhabited. The type checker
*is* the proof checker. These are not analogies - they are the same object.

| Mathematical concept  | In this code                                            |
|-----------------------|---------------------------------------------------------|
| Type `A`              | A `Val` with tag `VL_PI`, `VL_NAT`, etc.                |
| Term `t : A`          | A `Term` that `infer` maps to a `Val` conv-equal to `A` |
| Proof of `P`          | A `Term` whose inferred type conv-equals `P`            |
| Definitional equality | `conv(u, v)` on two evaluated `Val`s                    |
| Normal form           | `nbe_quote(nbe_eval(t))`                                |

The standard library (`code/lang/lib/`) demonstrates this concretely: the same
file contains `plus` (a program that computes) and `plus_comm` (a proof that
`plus m n = plus n m`). They are written in the same language, run through the
same evaluator, and checked by the same checker. See `lang/lib/README.md` for
a detailed walkthrough.



### What the system can do (quick tour)

*Arithmetic and recursion:*
```
:let rec fact : Nat → Nat =
  \n. boolrec (\b. Nat) (succ zero) (mult n (fact (pred n))) (iszero n)
fact (succ (succ (succ (succ zero))))    -- 4! = 24
```

*Length-indexed vectors:*
```
data Vec (A : Type) : Nat → Type where
  vnil  : Vec A zero
  vcons : Π(n : Nat). A → Vec A n → Vec A (succ n)
```
`head` on an empty vector is a type error, not a runtime crash.

*Proofs that compute:*
```
plus_comm (succ (succ zero)) (succ (succ (succ zero)))
-- → refl (succ (succ (succ (succ (succ zero)))))
```
The proof of commutativity *runs* and reduces to `refl`. Both sides compute
to 5.

*Universe polymorphism:*
```
:let id : Π(ℓ : Level). Π(A : Type_ℓ). A → A = \ℓ A x. x
id _ Nat zero        -- infers ℓ and A automatically
id _ Type Nat        -- works at a higher universe too
```

*Implicit arguments:*
```
id _ zero            -- A inferred as Nat
id _ true            -- A inferred as Bool
```



### How to navigate this guide

*Start with `02.md`.* It traces one concrete expression through
every layer of the system - parse, type-check, normalise, print - showing which
function does what and why. Read this before opening any source file.

Then read the source files in this order:

1. `code/core/term.h` - all `TM_` and `VL_` tags; skim once, return as reference
2. `code/core/eval.c` - `nbe_eval`, `nbe_vapp`, the eliminators
3. `code/core/check.c` - `infer`, `check`, `conv`
4. `code/core/parse.c` - `parse_term`, `parse_atom`, de Bruijn conversion
5. `code/core/elab.c` - hole filling (only relevant when you write `_`)

The mathematical background for every type former is in `code/core/MLTT.md`.



### Key vocabulary

*de Bruijn index* - A variable as a number: how many binders separate it
from its binding site. `\x. \y. x` → index 1 for `x`. Terms use indices;
values use *levels* (counted from the outside).

*NbE (Normalisation by Evaluation)* - Evaluate a `Term` into a `Val` using
the host language's function application, then `quote` back. No explicit
substitution needed during evaluation.

*Neutral value* - A value that is stuck: headed by a free variable (or axiom)
so no reduction rule can fire. `VL_NEUTRAL(lvl, spine)` represents `x` at
de Bruijn level `lvl` with eliminators `spine` accumulated on it.

*Bidirectional typing* - `infer` synthesises a type for most terms.
`check` verifies a term against a given type, and handles the terms that
*cannot* be inferred without context: `TM_LAM`, `TM_PAIR`, `TM_INL`,
`TM_INR`, `TM_SUP`.

*Spine* - The sequence of eliminators accumulated on a neutral head.
`natrec P z s x` with `x` a variable → neutral head `x` with spine
`[SP_NATREC(P,z,s)]`. Quoting replays the spine back to syntax.
