# Lark runtime — running compiled Lark on a Pico 2 / 2W

This directory turns a Lark program compiled to RISC-V (`program.S`) into a
firmware image for the **Raspberry Pi Pico 2 / 2W (RP2350)** and provides the small
C runtime the compiled code calls into.

```
runtime.c         heap (bump allocator), strings, float ops, show/convert builtins
runtime.h         the ABI the compiled code expects (heap layout, lark_ptr, …)
platform_pico.c   I/O for the Pico: print/read over USB-CDC (Pico SDK)
platform_posix.c  I/O for a host build (testing)
lark_entry.c      C entry point: calls __global_init__() then lark_main(io)
CMakeLists.txt    the Pico SDK build (board pico2, platform rp2350-riscv)
program.S         the compiled Lark program (generated; not committed)
```

The compiler emits two symbols the runtime links against: `lark_main` (your
program's `main`) and, optionally, `__global_init__` (top-level `let`s).

## Prerequisites

The Raspberry Pi **Pico SDK 2.2.0** and a **RISC-V** toolchain, installed under
`~/.pico-sdk`. The easiest way to get exactly this layout is the
[Pico VS Code extension](https://github.com/raspberrypi/pico-vscode), which
installs:

```
~/.pico-sdk/sdk/2.2.0/                              the SDK
~/.pico-sdk/toolchain/RISCV_ZCB_RPI_2_2_0_3/bin/   riscv32-unknown-elf-gcc
~/.pico-sdk/picotool/2.2.0-a4/picotool/picotool    flashing tool
~/.pico-sdk/cmake/.../bin/cmake  and  ~/.pico-sdk/ninja/.../ninja
```

`CMakeLists.txt` picks these up automatically via `$HOME/.pico-sdk`. If `cmake`,
`ninja`, and `picotool` are not already on your `PATH`, add them, e.g.:

```sh
export PATH="$HOME/.pico-sdk/cmake/v3.31.5/CMake.app/Contents/bin:\
$HOME/.pico-sdk/ninja/v1.12.1:\
$HOME/.pico-sdk/picotool/2.2.0-a4/picotool:$PATH"
```

## Build and flash

The simplest path is from the parent directory (`../`):

```sh
make pico  FILE=samples/03_primes.lark   # → runtime/build/lark_pico.uf2
make flash FILE=samples/03_primes.lark   # build + flash (board in BOOTSEL)
```

Equivalently, by hand from here:

```sh
python3 ../src/asm.py ../samples/03_primes.lark program.S
cmake -S . -B build -G Ninja -DPICO_BOARD=pico2 -DPICO_PLATFORM=rp2350-riscv
cmake --build build
picotool load -x build/lark_pico.uf2          # or drag build/lark_pico.uf2 onto the RP2350 drive
```

## Seeing output

Programs print over **USB-CDC serial**, and `platform_pico.c` waits for a host to
connect before the first line — so open a terminal after the board boots:

```sh
ls /dev/tty.usbmodem*               # find the port
screen /dev/tty.usbmodem* 115200    # Ctrl-A K to quit  (baud is ignored over USB-CDC)
```

## A note on hardware correctness

Real RISC-V (the RP2350's Hazard3 cores) traps on misaligned word access and is
strict about formatting, where a software VM is forgiving. The runtime and the
assembler therefore keep heap objects and `.rodata` strings word-aligned, and
`__show_float` trims trailing zeros so a Pico prints `8.5` (not `8.500000`),
matching every other backend. `build/` is regenerated by the build and is not
committed (it also embeds local SDK paths).
